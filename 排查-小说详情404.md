# 小说详情页 404（「小说不存在或已被下架」）排查步骤

本说明用于在**两台 API 服务器**和**两台数据库服务器**上执行固定命令，收集真实状态，定位根因，不做猜测。

---

## 前置条件

- 能 SSH 到跳板机：`ssh root@107.174.174.123`
- 从跳板机能 SSH 到：`10.50.0.2`、`10.50.0.3`、`10.50.0.4`、`10.50.0.5`
- 问题复现：访问 `https://younov.com/novel/nifeng-fanpan` 显示「小说不存在或已被下架」

---

## 一、在跳板机 107 上准备脚本

若仓库已包含排查脚本，在 107 上拉取：

```bash
cd /tmp/younov-build
git pull
```

把 API 机需要的脚本拷到两台 API（在 107 上执行一次）：

```bash
# 把 API 排查脚本拷到两台机
for ip in 10.50.0.2 10.50.0.3; do
  scp -o StrictHostKeyChecking=no /tmp/younov-build/scripts/check-api-server.sh root@$ip:/tmp/
  ssh -o StrictHostKeyChecking=no root@$ip "mkdir -p /opt/younov/api/scripts"
  scp -o StrictHostKeyChecking=no /tmp/younov-build/work/api/scripts/check-novel-in-db.js root@$ip:/opt/younov/api/scripts/
done
```

若仓库暂无这些文件，需从本机上传到 107 再按上路径分发。

**在 107 上经 Nginx 请求一次（与浏览器同路径）：**

```bash
curl -sS -o /dev/null -w "经 Nginx GET /api/novels/nifeng-fanpan => HTTP %{http_code}\n" "http://127.0.0.1/api/novels/nifeng-fanpan"
# 若 Nginx 在 107 上且监听 80，上面会轮询到 10.50.0.2 或 10.50.0.3
```

---

## 二、API 服务器 10.50.0.2

从跳板机执行：

```bash
ssh -o StrictHostKeyChecking=no root@10.50.0.2
```

在 **10.50.0.2** 上依次执行（可直接复制整段）：

```bash
# 1) 若脚本在 107 上，可先 scp 到本机（在 107 上执行一次）
# scp /tmp/younov-build/scripts/check-api-server.sh root@10.50.0.2:/tmp/

# 2) 在 10.50.0.2 上执行 API 检查脚本（若脚本在 /tmp）
bash /tmp/check-api-server.sh 2>&1

# 3) 本机请求小说详情（看 HTTP 状态码）
curl -sS -o /dev/null -w "GET /api/novels/nifeng-fanpan => HTTP %{http_code}\n" "http://127.0.0.1:8080/api/novels/nifeng-fanpan"

# 4) 用本机 DATABASE_URL 查库中是否有该书（必须在 API 目录）
cd /opt/younov/api && node scripts/check-novel-in-db.js nifeng-fanpan

# 5) 查看 .env 中 PEER_API_URL、DATABASE_URL 是否配置（不打印密码值）
grep -E '^PEER_API_URL=|^DATABASE_URL=' /opt/younov/api/.env 2>/dev/null | sed 's/=.*/=***/' || echo "无 .env 或无该变量"
```

把**完整终端输出**保存下来（例如存为 `check-10.50.0.2.txt`）。

ssh -o StrictHostKeyChecking=no root@10.50.0.3

在 **10.50.0.3** 上执行与 10.50.0.2 **相同的命令**（把上面 10.50.0.2 的整段命令再执行一遍），仅主机名不同。  
把**完整终端输出**保存（例如 `check-10.50.0.3.txt`）。

## 四、数据库服务器 10.50.0.4（主库）

从跳板机执行：

```bash
ssh -o StrictHostKeyChecking=no root@10.50.0.4
```

在 **10.50.0.4** 上执行：

```bash
# 1) PostgreSQL 监听端口
ss -tlnp | grep -E ':5432|:6432' || netstat -tlnp 2>/dev/null | grep -E ':5432|:6432'

# 2) 若有 psql 且知道库名/用户/密码，可直接查（把 younov_app / younov / 5432 换成实际值）
# PGPASSWORD=你的密码 psql -h 127.0.0.1 -p 5432 -U younov_app -d younov -t -c 'SELECT id, slug, title, "isPublished" FROM "Novel" WHERE slug = '\''nifeng-fanpan'\'';'
```

若无法在 DB 机上执行 psql，可跳过；**二、三**中在 API 机上执行的 `node scripts/check-novel-in-db.js` 已能说明「该 API 连的库里有没有这本书」。

## 六、根据结果判断

把四台机（或至少两台 API）的输出汇总后对照下表：

| 现象 | 可能原因 |
|------|----------|
| 两台 API 上 `curl 127.0.0.1:8080/api/novels/nifeng-fanpan` 都返回 **404** | 两台 API 连的库里都没有该书；或路由/代码未生效（见下） |
| 仅一台返回 **200**，另一台 **404** | 双库/双 API 数据不一致；或一台未配 `PEER_API_URL`，无法向对端回退 |
| 两台 API 上 `check-novel-in-db.js` 都显示 **存在**，但 curl 仍 404 | 代码或路由问题：检查 `src/index.js` 是否包含 `fetchFromPeer` 和 `/api/novels/:slug` 回退逻辑，以及 Nginx 是否把 `/api/` 正确反代到 8080 |
| 两台 API 上 `check-novel-in-db.js` 都显示 **不存在** | 当前 API 使用的 DATABASE_URL 指向的库里确实没有该书（需检查主/从库数据或导入） |
| 两台 API 的 `.env` 中 **没有 PEER_API_URL** | 本机无数据时不会向对端请求，轮询到「无数据」的那台就会 404 |

## 七、需要收集的文件清单（便于进一步对照代码）

在**两台 API 机**上建议查看（可把内容或 `cat` 输出保存）：

| 机器 | 路径 | 说明 |
|-----|------|------|
| 10.50.0.2 | `/opt/younov/api/.env` | 环境变量（可打码密码后保存） |
| 10.50.0.2 | `/opt/younov/api/src/index.js` | 前 50 行 + 含 `novels/:slug`、`fetchFromPeer`、`PEER_API_URL` 的片段 |
| 10.50.0.3 | 同上 | 同上 |

命令示例（在各自 API 机上）：

```bash
head -50 /opt/younov/api/src/index.js
grep -n "PEER_API_URL\|fetchFromPeer\|novels/:slug" /opt/younov/api/src/index.js
```

---

执行完上述步骤后，把两台 API 的检查输出（及若有的 DB 机输出）贴出来，即可根据真实状态继续定位根因，而不是猜测。
