# 轮询与回退机制说明

你的要求：**不论是数据库服务器还是 API 服务器，使用轮询机制，先判断正确的数据在哪台服务器上，找到正确的服务器再返回给前端。**

当前 API 代码已经按这个思路实现，流程如下。

---

## 当前实现（API 层：先本机再对端）

- **数据**在你这边的**两台数据库服务器**上；每台 **API** 通过 `DATABASE_URL` 连**其中一台**数据库。
- **前端**的请求经 Nginx 轮询到**任意一台 API**（10.50.0.2 或 10.50.0.3）。

**每次请求的流程：**

1. 请求落到 **API A**（例如 10.50.0.2）。
2. **API A** 用本机的 `DATABASE_URL` 查**自己连的那台数据库**。
3. **若本机库有数据** → 直接返回给前端（不再问对端）。
4. **若本机库无数据**（或查不到小说）→ API A 再请求**对端 API B**（10.50.0.3），并带上头 `x-internal-fallback: 1`，避免对端再回退造成环。
5. **API B** 收到后只查**自己的库**（不再次回退），把结果返回给 A。
6. **若 B 有数据** → A 把 B 的结果原样返回给前端。
7. **若 B 也无数据** → A 把本机的空结果返回给前端。

也就是说：**先判断“正确的数据”在本机还是对端，找到有数据的那一侧再返回给前端**，不会盲目返回空。

涉及的接口：

- `/api/leaderboard`（首页 Featured / Trending）
- `/api/novels`（Library 列表）
- `/api/free-novels`（Free 列表）

---

## 你需要做的配置（不是到 API 上“找数据”）

数据资源在你两台**数据库服务器**上，API 只负责**连对库**和**互相回退**。需要保证：

### 1. 两台 API 互相指对端（PEER_API_URL）

- 在 **10.50.0.2** 的 `/opt/younov/api/.env` 里：  
  `PEER_API_URL=http://10.50.0.3:8080`
- 在 **10.50.0.3** 的 `/opt/younov/api/.env` 里：  
  `PEER_API_URL=http://10.50.0.2:8080`

这样“本机无数据时”才会去问对端，实现“先判断在哪台再返回”。

### 2. 至少一台 API 连到“有 38 本”的数据库

- 若 **38 本在其中一台数据库服务器**上：  
  至少让**一台 API** 的 `DATABASE_URL` 指向这台数据库（主机、端口、库名、账号密码都指到有 38 本的那个库）。
- 另一台 API 可以连另一台库（或同一台库），无所谓；只要有一侧有数据，回退机制就会把**有数据的那一侧**的结果返回给前端。

### 3. Nginx 轮询到两台 API

107 上 Nginx 的 `upstream` 里保留两台 API，例如：

```nginx
upstream younov_api {
    server 10.50.0.2:8080;
    server 10.50.0.3:8080;
    keepalive 32;
}
```

这样前端请求会轮询到两台 API；无论落到哪一台，都会按上面流程“先本机再对端”，找到有数据的那台再返回。

---

## 小结

| 层级       | 作用 |
|------------|------|
| 数据库服务器 | 存真实数据（38 本）；API 通过 DATABASE_URL 连接，不在这里“找数据”，只保证连对库。 |
| API 服务器   | 先查本机库，无数据再请求对端 API，**先判断正确的数据在哪台（本机还是对端），再返回给前端**。 |
| Nginx       | 对两台 API 做轮询；无需判断“哪台有数据”，由 API 内部回退完成。 |

所以：**数据资源在数据库服务器上**；在 API 上不用“找数据”，只要配好 `DATABASE_URL` 和 `PEER_API_URL`，当前逻辑就会自动“先判断在哪台、再返回有数据的那一侧”给前端。
